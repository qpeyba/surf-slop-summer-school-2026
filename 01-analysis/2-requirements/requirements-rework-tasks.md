# Задачи на доработку требований

## Major

### TASK-REQ-001. Уточнить NFR-1 и развести с NFR-3

| Поле | Значение |
|------|----------|
| **Источник** | FINDING-REQ-001 |
| **Severity** | Major |
| **Файлы** | `non-functional-requirements.md` |
| **Проблема** | NFR-1 и NFR-3 пересекаются по смыслу. NFR-1 — частный случай NFR-3, но сформулированы как два независимых требования. |
| **Действие** | Переформулировать NFR-1 так, чтобы он говорил только о гарантии атомарности бронирования («0 двойных броней»), убрать из него общие слова про «источник истины» (это зона NFR-3). NFR-3 оставить как архитектурный принцип «клиент потребляет, не создаёт/не редактирует». |
| **Критерий готовности** | NFR-1 и NFR-3 не пересекаются по формулировкам, граница между ними очевидна. |
| **Риск, если не исправить** | При изменении одного NFR возникает несогласованность с другим; downstream-артефакты (API, ТЗ) могут по-разному интерпретировать границу ответственности клиента и бэкенда. |

## Minor

### TASK-REQ-002. Добавить UC-5 «Просмотр истории броней»

| Поле | Значение |
|------|----------|
| **Источник** | FINDING-REQ-002 |
| **Severity** | Minor |
| **Файлы** | `use-cases.md` |
| **Проблема** | FR-2.4 (история броней за 3 месяца) не имеет соответствующего UC. Сценарий самостоятелен и служит предусловием для UC-4 (оценка шефа). |
| **Действие** | Добавить UC-5: актёр Клиент, предусловия — есть завершённые брони, триггер — переход в историю, основной сценарий — просмотр списка → открытие карточки → опционально оценка шефа. |
| **Критерий готовности** | FR-2.4 покрыт use case'ом, связка «история → завершённая бронь → оценка шефа» явно описана в UC-5 со ссылкой на UC-4. |
| **Риск, если не исправить** | При проектировании экрана истории броней не будет формального сценария использования; связка с оценкой шефа останется неявной. |

### TASK-REQ-004. Добавить кросс-ссылки US ↔ UC/FR

| Поле | Значение |
|------|----------|
| **Источник** | FINDING-REQ-004 |
| **Severity** | Minor |
| **Файлы** | `user-stories.md`, `use-cases.md` |
| **Проблема** | US не ссылаются на соответствующие UC и FR; UC не ссылаются на покрываемые FR. |
| **Действие** | В `user-stories.md`: добавить столбец «Связанные FR/UC» с перекрёстными ссылками. В `use-cases.md`: добавить строку «Покрываемые требования» в каждый UC. |
| **Критерий готовности** | Из каждой US можно перейти к соответствующему FR/UC, из каждого UC — к покрываемым FR. |
| **Риск, если не исправить** | Снижение навигационной связности при трассировке требований → дизайн-бриф → ТЗ. |

### TASK-REQ-006. Зафиксировать в ТЗ, что бизнес-правила не хардкодятся

| Поле | Значение |
|------|----------|
| **Источник** | FINDING-REQ-006 |
| **Severity** | Info |
| **Файлы** | Не требуют изменений на уровне требований. Актуально для `5-mobile-app-spec/`. |
| **Проблема** | Значения «12 часов», «7 дней», «3 месяца» захардкожены в тексте FR. |
| **Действие** | На уровне требований оставить как есть (значения из domain-description). На шаге 6 (ТЗ экранов) явно указать, что эти значения получаются из ответа API/конфигурации. |
| **Критерий готовности** | В ТЗ экранов (шаг 6) для каждого бизнес-правила указан источник значения (API-ответ / конфигурация). |
| **Риск, если не исправить** | Разработчик захардкодит значения в клиенте; при изменении на бэкенде потребуется пересборка приложения. |
