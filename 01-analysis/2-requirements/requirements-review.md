### 2. Отсутствие неподтверждённых допущений — OK

Все формулировки требований выводятся из domain-description.md. Ни одной выдуманной функции, интеграции или бизнес-правила не обнаружено.

### 3. Разделение BR / FR / NFR / UC / US — OK

Функциональные и нефункциональные требования не смешаны. BR задают цели бизнеса, FR — поведение системы, NFR — ограничения и качества. US сформулированы в каноническом формате «Как / я хочу / чтобы». UC содержат актёров, предусловия, основной и альтернативные сценарии.

### 4. Наличие стабильных ID — OK

Все элементы имеют префиксы (`BR-*`, `FR-*`, `NFR-*`, `UC-*`, `US-*`). Нумерация стабильна — допускаются пропуски при сужении скоупа.

### 5. Проверяемость формулировок — OK

Все FR проверяемы — содержат конкретные действия и ожидаемые результаты. Все NFR — бинарные ограничения (соблюдено/не соблюдено). BR сформулированы как бизнес-цели.

### 6. Наличие источника у каждого требования — OK

Каждое требование содержит ссылку на конкретный раздел и строки domain-description.md.

### 7. Отсутствие противоречий между BR, FR, NFR, UC и US — OK

- Правило 12 часов: FR-4.1 → UC-2 (основной + альтернативный сценарий 4а) — согласовано
- Правило 10 минут: FR-3.1 → UC-1 (исключение 3а) — согласовано
- Перенос одним действием: FR-4.2 → UC-3 — согласовано
- NFR-3 (клиент потребляет, не создаёт) → все UC отправляют запросы к API, не содержат клиентской бизнес-логики — согласовано
- Все US имеют соответствие в FR — проверено

### 8. Отсутствие «висящих» требований — OK

Каждое требование используется как минимум в одном UC или US.

## Находки

### FINDING-REQ-001 — NFR-1 и NFR-3 частично дублируются

| Поле | Значение |
|------|----------|
| **Severity** | Major |
| **Где** | `non-functional-requirements.md` |
| **Проблема** | NFR-1 («Гарантия 0 двойных броней на стороне бэкенда») и NFR-3 («Вся бизнес-логика и проверки выполняются на бэкенде») пересекаются по смыслу. NFR-1 — частный случай NFR-3. |
| **Почему важно** | При изменении одного NFR может возникнуть несогласованность с другим. Разработчик или ревьюер не понимает, где искать каноническое утверждение об архитектурной ответственности. |
| **Рекомендация** | Либо объединить NFR-1 в NFR-3 как его уточнение, либо развести явно: NFR-3 = архитектурный принцип «клиент не создаёт/не редактирует слоты, программы, шефов», NFR-1 = конкретная гарантия атомарности бронирования. Второй вариант предпочтительнее. |

### FINDING-REQ-002 — UC не покрывают сценарии истории броней и лояльности

| Поле | Значение |
|------|----------|
| **Severity** | Minor |
| **Где** | `use-cases.md` |
| **Проблема** | FR-2.3 (балльный баланс и статус лояльности) и FR-2.4 (история броней за 3 месяца) не имеют соответствующих UC. Эти сценарии достаточно самостоятельны: клиент заходит в профиль → видит историю → находит завершённую бронь → оценивает шефа (связка с UC-4). |
| **Почему важно** | Без UC для истории броней неочевидна связка между просмотром завершённой брони и оценкой шефа (UC-4). |
| **Рекомендация** | Добавить UC-5 «Просмотр истории броней» — покрывает FR-2.4 и служит предусловием для UC-4. Для FR-2.3 достаточно ссылки из профиля, отдельный UC не обязателен. |

### FINDING-REQ-003 — Первичная ссылка FR-5.1 ведёт не в тот раздел

| Поле | Значение |
|------|----------|
| **Severity** | Minor |
| **Где** | `functional-requirements.md`, FR-5.1 |
| **Проблема** | Первичная ссылка на источник — «Раздел 4.1 (строка 118)», который описывает процесс записи и лишь упоминает push-напоминания. Канонический источник для уведомлений — Раздел 4.7 (таблица, строка 160). |
| **Почему важно** | При трассировке аналитик пойдёт в 4.1, а не в 4.7 — потеря навигационной связности. |
| **Рекомендация** | Поменять порядок ссылок: Раздел 4.7 (строка 160) сделать первым. |

### FINDING-REQ-004 — Отсутствуют кросс-ссылки между артефактами слоя

| Поле | Значение |
|------|----------|
| **Severity** | Minor |
| **Где** | `user-stories.md`, `use-cases.md` |
| **Проблема** | US и UC не содержат обратных ссылок друг на друга и на FR. Например, US-8 (отмена брони) не ссылается на UC-2, US-10 (оценка шефа) не ссылается на UC-4. |
| **Почему важно** | Снижает навигационную связность при трассировке требований → экраны → ТЗ. |
| **Рекомендация** | Добавить в каждую US ссылку на соответствующий UC (и/или FR). В каждый UC — ссылку на покрываемые FR. |

### FINDING-REQ-005 — NFR-2 не уточняет минимальную версию Android

| Поле | Значение |
|------|----------|
| **Severity** | Minor |
| **Где** | `non-functional-requirements.md`, NFR-2 |
| **Проблема** | NFR-2: «Android как нативное приложение» — без указания минимального API level. |
| **Почему важно** | Разработчику нужно знать, до какой версии поддерживать обратную совместимость. |

### FINDING-REQ-006 — Бизнес-правила захардкожены в требованиях без указания источника значения

| Поле | Значение |
|------|----------|
| **Severity** | Info |
| **Где** | `functional-requirements.md` (12ч, 7 дней, 3 месяца) |
| **Проблема** | Значения «12 часов», «7 дней», «3 месяца» зафиксированы прямо в тексте требований. Фреймворк (раздел 3, Шаг 6) рекомендует: «числа не хардкодятся — цена, лимиты, доступность приходят из ответов API». На уровне требований это допустимо (они взяты из domain-description), но в ТЗ экранов эти значения должны быть указаны как получаемые из API/конфигурации. |
| **Почему важно** | При изменении бизнес-правила на бэкенде клиент не должен требовать пересборки. |
| **Рекомендация** | На уровне требований оставить как есть (значения из domain-description). При написании ТЗ экранов (шаг 6) явно указать: значения получаются из ответа API, не хардкодятся. |

### FINDING-REQ-007 — NFR не содержат метрик производительности

| Поле | Значение |
|------|----------|
| **Severity** | Info |
| **Где** | `non-functional-requirements.md` |
| **Проблема** | Отсутствуют NFR на время загрузки расписания, поведение при потере сети, время отклика на действия пользователя. |
| **Почему важно** | Для мобильного приложения это стандартные ожидания, даже если заказчик их не сформулировал явно. Фреймворк (раздел 4.10) относит «NFR без метрики» к Major. |
| **Рекомендация** | Источник (domain-description) этих метрик не содержит — добавлять их без подтверждения заказчика нельзя (нарушение принципа защиты от галлюцинаций). Решение: на шаге проектирования ТЗ зафиксировать разумные значения по умолчанию и согласовать с заказчиком, либо осознанно оставить без метрик с пометкой «не входило в бриф». |
