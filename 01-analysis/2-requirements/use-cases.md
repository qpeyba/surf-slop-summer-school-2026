# Сценарии использования (Use Cases - UC)

### UC-1: Запись клиента на кулинарный класс
*   **Актёр**: Клиент.
*   **Предусловия**: Клиент авторизован в мобильном приложении. В профиле клиента заполнен список пищевых аллергий (при наличии).
*   **Триггер**: Клиент выбирает класс в расписании и нажимает кнопку записи.
*   **Основной сценарий**:
    1.  Клиент открывает приложение и видит расписание классов на 7 дней (с фильтрацией по умолчанию на текущую дату).
    2.  Клиент нажимает на карточку интересующего класса и изучает информацию (фото, меню, имя шефа, рейтинг шефа, уровень сложности, цена, адрес, свободные места).
    3.  Клиент нажимает кнопку «Записаться».
    4.  Приложение предлагает выбрать экипировку: «Своя экипировка» или «Прокатная экипировка (бесплатно)» (по умолчанию подтягивается флаг экипировки и список аллергий из профиля).
    5.  Клиент подтверждает выбор и нажимает «Подтвердить запись».
    6.  Приложение отправляет запрос бронирования на API бэкенда.
    7.  Бэкенд атомарно проверяет наличие свободных мест, резервирует место, создает бронь в статусе «Активна» и возвращает успешный ответ.
    8.  Приложение отображает экран успешного бронирования, бронь добавляется в историю клиента, счетчик свободных мест класса уменьшается.
*   **Альтернативные сценарии / Исключения**:
    *   *Исключение 3а: До начала класса осталось 10 минут и менее:* Кнопка записи неактивна, запись заблокирована (Раздел 2.1, строка 36).
    *   *Исключение 7а: Бэкенд сообщает, что мест больше нет:* Приложение выводит сообщение «Мест нет», запись отклоняется, интерфейс обновляет статус класса (Раздел 2.1, строка 40; Раздел 4.1, строка 116).
*   **Приоритет**: Высокий (Must Have)
*   **Ссылка на источник**: Раздел 2.1 (строки 33–36, 40), Раздел 2.3 (строки 60, 64), Раздел 2.4 (строки 76, 80), Раздел 4.1 (строки 112–120).

### UC-2: Отмена брони клиентом
*   **Актёр**: Клиент.
*   **Предусловия**: У клиента есть активная бронь на класс в статусе «Активна».
*   **Триггер**: Клиент нажимает кнопку «Отменить» в карточке активной брони.
*   **Основной сценарий**:
    1.  Клиент открывает список своих броней и выбирает активную запись.
    2.  Клиент нажимает кнопку «Отменить».
    3.  Приложение рассчитывает время, оставшееся до начала класса.
    4.  *Условие: До начала класса более 12 часов.* Приложение запрашивает подтверждение отмены (с указанием, что отмена бесплатная).
    5.  Клиент подтверждает отмену.
    6.  Приложение отправляет запрос на бэкенд.
    7.  Бэкенд переводит бронь в статус «Отменена клиентом», освобождает место на классе и возвращает успех.
    8.  Приложение информирует об успешной отмене, бронь меняет статус на «Отменена клиентом».
*   **Альтернативные сценарии / Исключения**:
    *   *Альтернативный сценарий 4а: До начала класса осталось 12 часов и менее (до 0 часов):*
        1. Приложение выводит предупреждение о том, что при отмене будет удержано 50% стоимости, и возврату подлежит только 50% (Раздел 2.4, строка 81; Раздел 4.2, строка 127).
        2. Клиент соглашается и нажимает «Подтвердить отмену».
        3. Приложение отправляет запрос на бэкенд.
        4. Бэкенд меняет статус брони на «Отменена клиентом», освобождает место и фиксирует сумму возврата (50%) в системе (Раздел 4.2, строка 128–129).
        5. Приложение показывает клиенту сообщение о том, что бронь отменена, а возврат 50% стоимости будет осуществлен владельцем вручную (Раздел 5, R11, строка 185).
*   **Приоритет**: Высокий (Must Have)
*   **Ссылка на источник**: Раздел 2.4 (строка 81), Раздел 4.2 (строки 124–130), Раздел 5 (R11, строка 185).

### UC-3: Перенос брони на другой класс
*   **Актёр**: Клиент.
*   **Предусловия**: У клиента есть активная бронь на класс в статусе «Активна».
*   **Триггер**: Клиент нажимает кнопку «Перенести» в карточке активной брони.
*   **Основной сценарий**:
    1.  Клиент переходит в детали своей активной брони и нажимает кнопку «Перенести».
    2.  Приложение открывает экран выбора доступных слотов на ближайшие 7 дней.
    3.  Клиент выбирает новый класс с доступными местами и подтверждает действие.
    4.  Приложение отправляет запрос переноса на API бэкенда.
    5.  Бэкенд в рамках одной транзакции (одним действием) отменяет старую бронь (с переводом в статус «Отмена клиентом» и освобождением места) и создает новую бронь на выбранный слот.
    6.  Приложение выводит уведомление об успешном переносе брони, старая бронь в списке закрывается, новая бронь отображается как «Активна».
*   **Альтернативные сценарии / Исключения**:
    *   *Исключение 5а: На новом выбранном слоте закончились места к моменту отправки запроса:* Бэкенд отклоняет транзакцию. Приложение выводит сообщение «На выбранном классе нет мест», перенос отклоняется, старая бронь остается активной.
*   **Приоритет**: Высокий (Must Have)
*   **Ссылка на источник**: Раздел 2.4 (строка 82), Раздел 4.3 (строки 133–136).

### UC-4: Выставление и редактирование оценки шефу
*   **Актёр**: Клиент.
*   **Предусловия**: Клиент посетил кулинарный класс. Бронь переведена в статус «Завершена».
*   **Триггер**: Клиент открывает завершенную бронь или переходит по предложению оценить шефа.
*   **Основной сценарий**:
    1.  Клиент открывает карточку завершенного класса в истории броней за последние 3 месяца.
    2.  Клиент нажимает кнопку «Оценить шефа».
    3.  Клиент выбирает количество звезд (от 1 до 5) и при желании вводит текстовый отзыв.
    4.  Клиент нажимает «Отправить».
    5.  Приложение отправляет оценку на бэкенд через API.
    6.  Бэкенд сохраняет оценку, пересчитывает средний рейтинг шефа и возвращает успех.
    7.  В приложении отображается сохраненная оценка.
*   **Альтернативные сценарии / Исключения**:
    *   *Альтернативный сценарий 2а: Клиент хочет отредактировать уже существующую оценку:*
        1. Клиент открывает завершенную бронь, где оценка уже выставлена (Раздел 2.4, строка 77; Раздел 4.6, строка 153).
        2. Клиент нажимает «Редактировать оценку».
        3. Изменяет количество звезд и/или текст отзыва и нажимает «Сохранить».
        4. Приложение отправляет измененные данные на бэкенд, рейтинг шефа обновляется.
*   **Приоритет**: Средний (Should Have)
*   **Ссылка на источник**: Раздел 2.4 (таблица, строка 77), Раздел 4.6 (строки 150–154).
