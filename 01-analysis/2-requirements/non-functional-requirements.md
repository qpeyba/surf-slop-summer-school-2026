# Нефункциональные требования (Non-functional Requirements - NFR)

| ID | Категория | Формулировка | Приоритет | Ссылка на источник (domain-description.md) |
|---|---|---|---|---|
| **NFR-1** | Надежность и целостность данных | Гарантия «0 двойных броней» должна обеспечиваться на стороне бэкенда посредством атомарной проверки свободных мест. Приложение должно полагаться на ответ API. | Высокий (Must Have) | Раздел 2.4 (строка 80), Раздел 4.1 (строки 115-116), Раздел 5 (R1, строка 175) |
| **NFR-2** | Платформа / Переносимость | Приложение должно быть реализовано только под мобильную платформу Android как нативное приложение и распространяться через Google Play. | Высокий (Must Have) | Раздел 5 (R9, строка 183) |
| **NFR-3** | Архитектура системы | Приложение должно выступать исключительно в качестве потребителя данных через API. Вся бизнес-логика и проверки (свободные места, двойные бронирования) выполняются на бэкенде, который является «black-box» источником истины. | Высокий (Must Have) | Раздел 5 (R1, R2, строки 175–176) |
| **NFR-4** | Ограничение скоупа поставок | Разрабатываемая система включает интерфейсы только для роли «Клиент». Интерфейсы шефа и администратора/владельца находятся вне скоупа данной поставки. | Высокий (Must Have) | Раздел 3 (таблица, строки 102–104), Раздел 5 (R3, строка 177) |
| **NFR-5** | Контракт данных | Обмен данными с бэкендом должен происходить по канонической схеме API без необходимости поддерживать исторические данные или легаси-контракты. | Высокий (Must Have) | Раздел 5 (R6, строка 180) |
