# Диаграммы взаимодействия — «Шеф-стол»

> Построены на основе сценариев использования (UC-1) и функциональных требований (FR-3.1, FR-3.2).  
> Описывают ключевые потоки взаимодействия «клиентское приложение → бэкенд».  
> Все критические проверки — на стороне бэкенда; приложение доверяет ответам API и корректно обрабатывает отказы.

---

## 1. createBooking — запись на класс (UC-1)

Клиент бронирует место на выбранный слот. Бэкенд атомарно проверяет доступность и резервирует место.

### Участники

| Участник | Роль |
|----------|------|
| **Client (App)** | Мобильное приложение клиента. Отправляет запрос бронирования и обрабатывает ответ |
| **Backend (API)** | Black-box бэкенд. Владелец данных, выполняет атомарную проверку и резервирование |
| **Client (UI)** | Пользовательский интерфейс приложения: экран подтверждения, экран ошибки |

### Предусловия

- Клиент авторизован (валидный токен в заголовке `Authorization`).
- Клиент выбрал слот из расписания и заполнил форму брони (тип экипировки).
- Приложение уже получило актуальные данные слота через `GET /slots/{slotId}` и отобразило свободные места.

---

### 1.1. Диаграмма — createBooking (ветки 201 / 409 / 410)

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant App as Клиентское приложение
    participant API as Бэкенд (API)

    rect rgb(240, 248, 255)
        note over U, API: Предварительный шаг: просмотр карточки класса
        U->>App: Открывает карточку класса
        App->>API: GET /slots/{slotId}
        API-->>App: 200 Slot { capacity, bookedCount, status, dateTime }
        App->>App: Свободно = capacity − bookedCount<br/>Доступен = status=Активен AND t−10мин > now
        App-->>U: Показывает карточку: меню, шеф,<br/>«Осталось N из M», кнопка «Записаться»
    end

    rect rgb(255, 250, 240)
        note over U, API: Шаг 1: заполнение формы брони
        U->>App: Нажимает «Записаться»
        App-->>U: Форма: экипировка (своя / прокат),<br/>аллергии (подтянуты из профиля)
        U->>App: Выбирает экипировку, нажимает «Подтвердить»
    end

    rect rgb(240, 255, 240)
        note over U, API: Шаг 2: запрос бронирования (атомарная проверка на бэкенде)
        App->>API: POST /bookings { slotId, equipmentType }
        activate API
        
        alt Успешное бронирование (201 Created)
            API->>API: ✓ Слот активен<br/>✓ Окно записи открыто (≥10 мин до начала)<br/>✓ Есть свободные места (bookedCount < capacity)
            API->>API: Атомарно: bookedCount++,<br/>создаёт Booking { status: "Активна" }
            API-->>App: 201 Created { booking }
            deactivate API
            App-->>U: Экран успеха:<br/>«Вы записаны!»<br/>Бронь в «Мои брони»
            App->>API: GET /slots/{slotId} (обновить свободные места)
            API-->>App: 200 Slot { bookedCount: +1 }

        else Мест нет (409 Conflict)
            API->>API: ✓ Слот активен, окно записи открыто<br/>✗ bookedCount == capacity
            API-->>App: 409 Conflict { error: "slot_full" }
            deactivate API
            App-->>U: Сообщение «Мест нет»<br/>Кнопка записи блокируется<br/>Счётчик: «Осталось 0 из M»
            App->>API: GET /slots/{slotId} (сверить доступность)
            API-->>App: 200 Slot { capacity, bookedCount }
            App->>App: Обновляет состояние: «Мест нет»

        else Слот недоступен (410 Gone)
            API->>API: ✗ slot.status == "Отменён студией"<br/>ИЛИ<br/>✗ now > dateTime − 10 мин
            alt Класс отменён студией
                API-->>App: 410 Gone { error: "slot_cancelled",<br/>reason: "Форс-мажор: …" }
            else Окно записи закрыто
                API-->>App: 410 Gone { error: "slot_started" }
            end
            deactivate API
            alt slot_cancelled
                App-->>U: Сообщение «Класс отменён студией»<br/>Запись недоступна
            else slot_started
                App-->>U: Сообщение «Запись закрыта»<br/>До начала осталось < 10 минут
            end
            App->>API: GET /slots/{slotId} (получить актуальный статус)
            API-->>App: 200 Slot { status: "Отменён студией" }
            App->>App: Обновляет карточку: слот неактивен
        end
    end
```

---

### 1.2. Сводка ответов API

| Код | Условие | Тело ответа | Действие приложения |
|-----|---------|-------------|---------------------|
| **201 Created** | Слот активен, места есть, окно записи открыто | `{ booking: { id, slotId, status: "Активна", ... } }` | Экран успеха, бронь в список «Мои брони», обновить `bookedCount` |
| **409 Conflict** | Слот активен, но `bookedCount == capacity` | `{ error: "slot_full" }` | Сообщение «Мест нет», блокировка кнопки записи, обновить счётчик |
| **410 Gone** | `slot.status == "Отменён студией"` | `{ error: "slot_cancelled", reason: "…" }` | Сообщение «Класс отменён студией», запись заблокирована |
| **410 Gone** | `now > dateTime − 10 мин` | `{ error: "slot_started" }` | Сообщение «Запись закрыта», кнопка неактивна |
| **401 Unauthorized** | Токен истёк / отсутствует | `{ error: "unauthorized" }` | Редирект на экран авторизации |

---

### 1.3. Примечания

1. **Атомарность на бэкенде.** Все проверки (статус слота, окно записи, наличие мест) выполняются в одной транзакции. Между проверкой `bookedCount < capacity` и инкрементом `bookedCount++` не может вклиниться другой запрос — это гарантирует «0 двойных броней» (R1).

2. **Pre-check в приложении.** Кнопка «Записаться» блокируется на клиенте, если:
   - `status != "Активен"` (слот отменён)
   - `now > dateTime − 10 мин` (окно записи закрыто)
   - `bookedCount >= capacity` (мест нет)
   Однако клиентская проверка — не гарантия; окончательное решение — за бэкендом (могло измениться между загрузкой слота и отправкой запроса).

3. **Идемпотентность.** Приложение должно использовать заголовок `Idempotency-Key` (уникальный на каждую попытку бронирования), чтобы исключить двойное создание брони при повторной отправке запроса из-за сетевой ошибки.

4. **После 409 / 410.** Приложение обновляет данные слота через `GET /slots/{slotId}`, чтобы синхронизировать UI с актуальным состоянием бэкенда (статус, свободные места).

## 2. cancelBooking — отмена брони клиентом (UC-2)

Клиент отменяет активную бронь. Правила возврата зависят от оставшегося до начала класса времени: >12ч — бесплатно, ≤12ч — возврат 50% с фиксацией `refundAmount` на бэкенде.

### Участники

| Участник | Роль |
|----------|------|
| **Client (App)** | Приложение. Рассчитывает время до начала, информирует о правилах возврата, отправляет запрос отмены |
| **Backend (API)** | Бэкенд. Переводит бронь в «Отменена клиентом», освобождает место, фиксирует `refundAmount` при возврате ≤12ч |
| **Client (UI)** | Интерфейс: карточка брони, предупреждение о сумме возврата, подтверждение отмены |

### Предусловия

- Клиент авторизован.
- У клиента есть активная бронь (`booking.status == "Активна"`).
- Приложение знает `booking.slotId` и `slot.dateTime` (получены при загрузке списка броней).

---

### 2.1. Диаграмма — cancelBooking (ветки 200 >12ч / 200 ≤12ч / 409)

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant App as Клиентское приложение
    participant API as Бэкенд (API)

    rect rgb(240, 248, 255)
        note over U, API: Предварительный шаг: карточка активной брони
        U->>App: Открывает «Мои брони» → активная бронь
        App->>API: GET /bookings/{bookingId}
        API-->>App: 200 Booking { status: "Активна", slotId, ... }
        App->>API: GET /slots/{slotId}
        API-->>App: 200 Slot { dateTime, price, ... }
        App->>App: remaining = slot.dateTime − now<br/>refundText = remaining > 12ч ? «Отмена бесплатна»<br/>: «Возврат 50% стоимости»
        App-->>U: Карточка брони: меню, дата, адрес,<br/>кнопка «Отменить»
    end

    rect rgb(255, 250, 240)
        note over U, API: Шаг 1: подтверждение отмены
        U->>App: Нажимает «Отменить»
        alt До начала > 12 часов (бесплатная отмена)
            App-->>U: «Отмена бесплатна.<br/>Подтвердить отмену?»
        else До начала ≤ 12 часов (возврат 50%)
            App-->>U: «При отмене возвращается 50% стоимости.<br/>Сумма к возврату: price / 2.<br/>Возврат осуществит владелец вручную.<br/>Подтвердить отмену?»
        end
        U->>App: Подтверждает отмену
    end

    rect rgb(240, 255, 240)
        note over U, API: Шаг 2: запрос отмены на бэкенд
        App->>API: POST /bookings/{bookingId}/cancel
        activate API

        alt Успешная отмена (200 OK)
            API->>API: Проверяет: бронь активна
            API->>API: Меняет статус: «Отменена клиентом»<br/>Освобождает место: bookedCount−−
            alt remaining > 12ч
                API->>API: refundAmount = null
            else remaining ≤ 12ч
                API->>API: refundAmount = price × 0.5
            end
            API-->>App: 200 OK { booking: { status: "Отменена клиентом",<br/>refundAmount: null | price×0.5 } }
            deactivate API
            alt refundAmount == null
                App-->>U: «Бронь отменена»
            else refundAmount > 0
                App-->>U: «Бронь отменена.<br/>К возврату: refundAmount.<br/>Возврат осуществит владелец вручную»
            end
            App->>App: Бронь → «Отменена клиентом» в списке

        else Бронь уже неактивна (409 Conflict)
            API->>API: ✗ booking.status != "Активна"
            API-->>App: 409 Conflict { error: "booking_not_active" }
            deactivate API
            App-->>U: «Бронь уже отменена или завершена»
            App->>API: GET /bookings/{bookingId} (сверить статус)
            API-->>App: 200 Booking { status }
            App->>App: Обновляет карточку брони

        else Бронь не найдена (404 Not Found)
            API-->>App: 404 Not Found { error: "booking_not_found" }
            deactivate API
            App-->>U: «Бронь не найдена»
        end
    end
```

---

### 2.2. Сводка ответов API

| Код | Условие | Тело ответа | Действие приложения |
|-----|---------|-------------|---------------------|
| **200 OK** | Бронь активна, отмена успешна, >12ч | `{ booking: { status: "Отменена клиентом", refundAmount: null } }` | Сообщение «Бронь отменена», обновить список |
| **200 OK** | Бронь активна, отмена успешна, ≤12ч | `{ booking: { status: "Отменена клиентом", refundAmount: price×0.5 } }` | Сообщение с суммой возврата, пометка «возврат — вручную» |
| **409 Conflict** | `booking.status != "Активна"` | `{ error: "booking_not_active" }` | Сообщение «Бронь уже отменена или завершена», обновить карточку |
| **404 Not Found** | `bookingId` не существует | `{ error: "booking_not_found" }` | Сообщение «Бронь не найдена» |

---

### 2.3. Примечания

1. **Pre-check времени.** Приложение вычисляет `remaining = slot.dateTime − now` до отправки запроса, чтобы показать правильное предупреждение (>12ч vs ≤12ч). Бэкенд при получении запроса пересчитывает оставшееся время самостоятельно и фиксирует `refundAmount` — это гарантирует корректность, даже если запрос «завис» на границе 12-часового окна.

2. **Атомарность на бэкенде.** Операции «смена статуса брони → `bookedCount−−` → расчёт `refundAmount`» выполняются в одной транзакции. Если бронь уже была отменена другим запросом, транзакция получит `409`.

3. **refundAmount — только фиксация.** Приложение отображает `refundAmount` как справочную информацию. Фактический возврат денег — ручной процесс владельца (наличные / перевод). Приложение не инициирует платёжных операций.

## 3. transferBooking — перенос брони на другой слот (UC-3)

Клиент переносит активную бронь на новый слот. Бэкенд выполняет операцию одной транзакцией: отменяет старую бронь и создаёт новую. При любом сбое на новом слоте старая бронь остаётся нетронутой.

### Участники

| Участник | Роль |
|----------|------|
| **Client (App)** | Приложение. Отображает доступные слоты, отправляет запрос переноса |
| **Backend (API)** | Бэкенд. Выполняет атомарную транзакцию «отмена + создание», проверяет доступность нового слота |
| **Client (UI)** | Интерфейс: экран выбора слота, подтверждение переноса, результат |

### Предусловия

- Клиент авторизован.
- У клиента есть активная бронь (`booking.status == "Активна"`).
- Приложение загрузило список доступных слотов на 7 дней (через `GET /slots`).

---

### 3.1. Диаграмма — transferBooking (ветки 200 / 409 / 410)

```mermaid
sequenceDiagram
    actor U as Пользователь
    participant App as Клиентское приложение
    participant API as Бэкенд (API)

    rect rgb(240, 248, 255)
        note over U, API: Предварительный шаг: карточка активной брони
        U->>App: Открывает «Мои брони» → активная бронь
        App->>API: GET /bookings/{bookingId}
        API-->>App: 200 Booking { id, slotId, status: "Активна", equipmentType }
        App-->>U: Карточка брони: меню, дата, шеф,<br/>кнопки «Отменить», «Перенести»
    end

    rect rgb(255, 250, 240)
        note over U, API: Шаг 1: выбор нового слота
        U->>App: Нажимает «Перенести»
        App->>API: GET /slots?dateFrom=now&dateTo=now+7d
        API-->>App: 200 [Slot] (только status="Активен" AND bookedCount < capacity)
        App-->>U: Список доступных слотов на 7 дней<br/>(текущий слот исключён)
        U->>App: Выбирает новый слот
        App-->>U: «Перенести бронь с [старый слот]<br/>на [новый слот]?»
        U->>App: Подтверждает перенос
    end

    rect rgb(240, 255, 240)
        note over U, API: Шаг 2: запрос переноса (одна транзакция)
        App->>API: POST /bookings/{bookingId}/transfer { newSlotId }
        activate API

        alt Успешный перенос (200 OK)
            API->>API: BEGIN TRANSACTION<br/>1. Старая бронь: status → «Отменена клиентом»,<br/>   oldSlot.bookedCount−−<br/>2. Новый слот: проверка bookedCount < capacity ✓<br/>3. Новый слот: bookedCount++<br/>4. Создаёт Booking { status: "Активна",<br/>   slotId: newSlotId, equipmentType: из старой }
            API->>API: COMMIT
            API-->>App: 200 OK { oldBooking: { status: "Отменена клиентом" },<br/>newBooking: { id, status: "Активна", slotId } }
            deactivate API
            App-->>U: «Бронь перенесена»<br/>Старая → «Отменена клиентом»<br/>Новая → «Активна» в списке

        else Новый слот заполнен (409 Conflict)
            API->>API: BEGIN TRANSACTION<br/>1. Старая бронь: status → «Отменена клиентом»,<br/>   oldSlot.bookedCount−−<br/>2. Новый слот: bookedCount == capacity ✗
            API->>API: ROLLBACK
            API-->>App: 409 Conflict { error: "target_slot_full" }
            deactivate API
            App-->>U: «На выбранном классе нет мест»<br/>Старая бронь остаётся активной
            App->>API: GET /bookings/{bookingId} (подтвердить, что старая не тронута)
            API-->>App: 200 Booking { status: "Активна" }

        else Новый слот недоступен (410 Gone)
            API->>API: BEGIN TRANSACTION<br/>1. Старая бронь: status → «Отменена клиентом»,<br/>   oldSlot.bookedCount−−<br/>2. Новый слот: status="Отменён студией"<br/>   ИЛИ now > dateTime − 10min ✗
            API->>API: ROLLBACK
            API-->>App: 410 Gone { error: "target_slot_unavailable" }
            deactivate API
            App-->>U: «Выбранный класс более недоступен»<br/>Старая бронь остаётся активной
            App->>API: GET /bookings/{bookingId}
            API-->>App: 200 Booking { status: "Активна" }
            App->>API: GET /slots (обновить список доступных)
            API-->>App: 200 [Slot]
        end
    end
```

---

### 3.2. Сводка ответов API

| Код | Условие | Тело ответа | Действие приложения |
|-----|---------|-------------|---------------------|
| **200 OK** | Транзакция успешна | `{ oldBooking: { status: "Отменена клиентом" }, newBooking: { id, status: "Активна", slotId } }` | Сообщение «Бронь перенесена», обновить список: старая закрыта, новая активна |
| **409 Conflict** | Новый слот полон (`target_slot_full`) | `{ error: "target_slot_full" }` | Сообщение «На выбранном классе нет мест», старая бронь не тронута |
| **410 Gone** | Новый слот недоступен (`target_slot_unavailable`) | `{ error: "target_slot_unavailable" }` | Сообщение «Выбранный класс более недоступен», старая бронь не тронута, обновить список слотов |
| **409 Conflict** | Старая бронь уже неактивна | `{ error: "booking_not_active" }` | Сообщение «Бронь уже отменена или завершена» |

---

### 3.3. Примечания

1. **Одна транзакция.** Бэкенд выполняет отмену старой и создание новой брони в рамках одной ACID-транзакции (`BEGIN → операции → COMMIT`). Если проверка нового слота не проходит — `ROLLBACK`, старая бронь остаётся в исходном состоянии (`"Активна"`). Это гарантирует, что клиент не потеряет место при неудачном переносе.

2. **Наследование атрибутов.** Новая бронь наследует `equipmentType` от старой. Приложение не отправляет повторно данные экипировки — они уже есть в исходной брони.

3. **Отсутствие правил возврата при переносе.** В отличие от `cancelBooking`, при переносе `refundAmount` не рассчитывается — старая бронь просто закрывается без возврата. Перенос не имеет временных ограничений (>12ч / ≤12ч), в отличие от отмены.

4. **Pre-check на клиенте.** Приложение исключает из списка слоты с `status != "Активен"` и `bookedCount >= capacity` ещё до отправки запроса. Финальная проверка — на бэкенде (место могли занять за время выбора).

## 4. upsertReview — выставление и редактирование оценки шефу (UC-4)

Клиент выставляет или редактирует оценку шефу (1–5 звёзд + опциональный текст) для завершённой брони. Операция — upsert: если оценки ещё нет — создаётся, если есть — обновляется. Бэкенд пересчитывает рейтинг шефа.

### Участники

| Участник | Роль |
|----------|------|
| **Client (App)** | Приложение. Отправляет оценку, отображает текущую (если есть) или форму |
| **Backend (API)** | Бэкенд. Сохраняет/обновляет `reviewRating` и `reviewText`, пересчитывает `Instructor.rating` |
| **Client (UI)** | Интерфейс: звёзды (1–5), поле отзыва, кнопка «Отправить» / «Сохранить» |

### Предусловия

- Клиент авторизован.
- Бронь находится в статусе `"Завершена"`.
- Приложение получило `booking.id` и `booking.slot.instructorId` (через `GET /bookings/{bookingId}`).

---

### 4.2. Сводка ответов API

| Код | Условие | Тело ответа | Действие приложения |
|-----|---------|-------------|---------------------|
| **200 OK** | Бронь завершена, оценка сохранена/обновлена | `{ review: { rating, text }, instructorRating: decimal }` | Сообщение «Оценка сохранена», обновить звёзды и текст в карточке |
| **403 Forbidden** | `booking.status != "Завершена"` | `{ error: "booking_not_completed" }` | Сообщение «Оценить можно только после завершения», обновить статус брони |
| **404 Not Found** | `bookingId` не существует | `{ error: "booking_not_found" }` | Сообщение «Бронь не найдена» |
| **400 Bad Request** | `rating` не в диапазоне 1–5 | `{ error: "invalid_rating" }` | Валидация на клиенте до отправки; если дошло — показать ошибку |

### 4.3. Примечания

1. **Upsert-семантика.** Один endpoint `PUT /bookings/{bookingId}/review` обрабатывает и создание (первая оценка), и редактирование (повторная оценка). Клиенту не нужно различать эти сценарии на уровне API — он всегда отправляет `{ rating, text }`.

2. **Пересчёт рейтинга шефа.** Бэкенд пересчитывает `Instructor.rating` как среднее арифметическое всех `reviewRating` по всем броням этого шефа. Новое значение возвращается в ответе, приложение может сразу отобразить его в карточке (или дождаться следующего `GET /slots/{slotId}`).

3. **Ограничение статусом.** Оценку можно выставить только для броней в статусе `"Завершена"`. Активные, отменённые и no-show брони не принимают оценку — бэкенд возвращает `403`. Приложение скрывает кнопку «Оценить» для всех статусов, кроме `"Завершена"`.

4. **Редактирование в любой момент.** Ограничений по времени на редактирование нет — клиент может изменить оценку и через день, и через месяц после класса. Приложение всегда показывает кнопку «Редактировать оценку» для завершённой брони с уже существующей оценкой.
