# Выбор стека для клиентского приложения — «Шеф-стол»

> На основе анализа `01-analysis/`. Бэкенд: Go + REST API (JWT). Клиент: Android, с нуля.

---

## 1. Исходные технические ограничения (из NFR и API)

| # | Ограничение | Значение |
|---|-------------|----------|
| 1 | Платформа | **Только Android**. Распространение через Google Play. minSdk — уточняется |
| 2 | Стиль API | **Только REST** (5 доменов OpenAPI, 12 endpoints) |
| 3 | Авторизация | OTP по SMS → JWT Bearer-токен. Хранение: EncryptedSharedPreferences/Keystore |
| 4 | Push-уведомления | Firebase Cloud Messaging (FCM). Запрос `POST_NOTIFICATIONS` (Android 13+) |
| 5 | Дизайн-система | Material Design 3 (Material You). 412×915 dp эталон |
| 6 | Бизнес-логика | Вся на бэкенде. Клиент — «тонкий» потребитель API |
| 7 | Офлайн-режим | Отсутствует. Все данные — из API |
| 8 | Платёжная интеграция | Отсутствует в MVP (заглушка «Скоро: онлайн-оплата») |
| 9 | Роли | Только «Клиент» |
| 10 | Состояния экранов | 4 состояния: loading (skeleton), empty, error, success |

## 2. Варианты стека

### Вариант A: Kotlin + Jetpack Compose (нативный Android, современный)

```
Язык:         Kotlin 2.x
UI:           Jetpack Compose + Material 3
Навигация:    Compose Navigation (type-safe routes)
Сеть:         Retrofit + OkHttp + kotlinx.serialization (или Moshi)
DI:           Hilt (Dagger) или Koin
State:        ViewModel + StateFlow (или MVI через Orbit/Decompose)
Хранение:     AndroidX Security (EncryptedSharedPreferences) + DataStore
Push:         Firebase Cloud Messaging
Изображения:  Coil (Compose-native загрузка фото)
Календарь:    Material 3 DatePicker (встроенный) или кастомный
```

**Плюсы:**
- Полное соответствие NFR-2 (нативный Android)
- Современный декларативный UI, меньше boilerplate
- Полная поддержка Material 3 из коробки
- Единая экосистема: AndroidX, Google Play Services для FCM
- Самый богатый доступ к Android API: EncryptedSharedPreferences, Keystore, POST_NOTIFICATIONS

**Минусы:**
- Только Android (нет iOS, даже в перспективе)
- Compose — относительно молод; могут быть граничные баги
- Требует знания Kotlin + корутин + Flow
- Сборка тяжелее, чем у кроссплатформы (полный Android-тулчейн)

**Итог:** Максимальная нативность, полный доступ к Android API, минимум компромиссов. Идеально, если iOS не нужен ни сейчас, ни в перспективе 2–3 лет.

### Вариант B: Kotlin + XML Views (нативный Android, классический)

```
Язык:         Kotlin 2.x
UI:           XML-вёрстка + ViewBinding/DataBinding + Material Components
Навигация:    Jetpack Navigation (XML-navGraph) или Cicerone
Сеть:         Retrofit + OkHttp + Moshi
DI:           Hilt / Koin
State:        ViewModel + LiveData / StateFlow
Хранение:     AndroidX Security + DataStore
Push:         Firebase Cloud Messaging
Изображения:  Glide / Picasso
Календарь:    MaterialDatePicker или CustomView
```

**Плюсы:**
- Стабильная, проверенная годами технология
- Огромная база знаний, StackOverflow, документации
- Предсказуемое поведение на всех версиях Android
- Легче найти разработчиков с опытом XML-вёрстки
- Полный доступ к Android API

**Минусы:**
- XML-вёрстка многословна (boilerplate)
- Сложные анимации и переходы делать труднее
- Material 3 через XML — неполная поддержка (Material Components всё ещё догоняет MD3)
- В долгосрочной перспективе Google двигает индустрию к Compose

**Итог:** Консервативный выбор. Подходит, если команда не знает Compose и хочет минимизировать риски. Но для нового проекта в 2026 году Compose выглядит предпочтительнее.

---

### Вариант C: Kotlin Multiplatform (KMP) + Compose Multiplatform

```
Язык:         Kotlin 2.x
UI:           Compose Multiplatform (общий UI для Android + iOS)
Навигация:    Decompose / Voyager / Compose Navigation (multiplatform)
Сеть:         Ktor Client + kotlinx.serialization
DI:           Koin (multiplatform) или ручной DI
State:        ViewModel (Android) + собственный ViewModel для iOS
Хранение:     AndroidX Security (Android) / NSUserDefaults+iOSKeychain (iOS expect/actual)
Push:         FCM (Android expect/actual) + APNs (iOS expect/actual)
Изображения:  Coil (Android) + Nuke (iOS) через expect/actual
Календарь:    Кастомный Compose-компонент
```

**Плюсы:**
- Общий бизнес-код (сеть, модели, логика) — пишется один раз
- Compose UI шарится между платформами (с оговорками)
- Задел на iOS «бесплатно» — можно запустить, когда потребуется
- Kotlin — современный язык, общий с бэкендом по экосистеме (Gradle)
- Ktor Client отлично работает с REST API

**Минусы:**
- Compose Multiplatform для iOS всё ещё в стадии стабилизации (хотя быстро зреет)
- Требуется expect/actual для платформенных API: FCM, Keystore, уведомления
- Навигация на KMP всё ещё фрагментирована (нет единого стандарта)
- Сложнее отладка, чем в чисто нативном Android
- Меньше разработчиков с KMP-опытом на рынке
- Зрелость: вы всё ещё на передовой, возможны breaking changes

**Итог:** Выбор, если iOS нужен в дорожной карте 1–2 года. Позволяет сэкономить на переписывании бизнес-логики, но несёт риски незрелости iOS-таргета Compose Multiplatform.

### Вариант D: Flutter

```
Язык:         Dart 3.x
UI:           Flutter Widgets + Material 3 (полная поддержка)
Навигация:    GoRouter (или встроенный Navigator 2.0)
Сеть:         Dio + json_serializable (или retrofit_generator)
DI:           GetIt / Riverpod / Provider
State:        Riverpod / BLoC / MobX
Хранение:     flutter_secure_storage (Keystore/Keychain)
Push:         firebase_messaging
Изображения:  cached_network_image
Календарь:    TableCalendar / встроенный DatePicker
```

**Плюсы:**
- Один код на Android + iOS + (возможно) Web
- Material 3 — полная поддержка из коробки во Flutter
- Быстрая разработка: hot reload, единая кодовая база
- Огромное сообщество, много готовых пакетов
- Не зависит от Android API напрямую — меньше платформенных заморочек
- Лёгкий вход для новичков

**Минусы:**
- **Противоречит NFR-2** — Flutter не является нативным Android-приложением (хотя компилируется в нативный ARM-код). NFR-2 говорит «нативное приложение», что обычно трактуется как Kotlin/Java + Android SDK.
- Не-нативный UI: Flutter рендерит свой Canvas, а не использует нативные Android Views. Может выглядеть «не по-андроидовски».
- Доступ к Android API — через Platform Channels (код на Kotlin/Java). FCM, Keystore, POST_NOTIFICATIONS — все требуют мостов.
- Размер APK больше (~5-8 MB сверху за счёт Flutter engine).
- Dart — отдельный язык, не используется на бэкенде (Go). Команде придётся учить ещё один язык.

**Итог:** Самый быстрый путь к двум платформам. Но прямое противоречие с NFR-2 (требуется пересмотр требования). Идеально, если iOS критичен **сейчас**, а не в перспективе.

## 4. Моя рекомендация

> **A: Kotlin + Jetpack Compose** — приоритетный вариант.

Причины:
1. **Соответствует NFR-2** без оговорок.
2. Compose — зрелое решение в 2026, Material 3 из коробки.
3. Все нужные Android API доступны напрямую (EncryptedSharedPreferences, FCM, POST_NOTIFICATIONS — без мостов).
4. Приложение относительно простое (7 экранов, 4 шторки) — Compose Navigation покрывает всю навигацию без лишних фреймворков.
5. Нет требования iOS. Если оно появится через 2-3 года — можно будет рассмотреть KMP-миграцию бизнес-слоя (сеть, модели, логика легко выносятся в shared-модуль), а UI оставить нативным.

**Но** — выбор стека сильно зависит от контекста команды. Далее — режим Q&A.
