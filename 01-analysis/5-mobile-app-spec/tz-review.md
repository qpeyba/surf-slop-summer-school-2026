- **LOGIC-002 (Доступность)**: формула `canBook` используется в SCR-001, SCR-002, SCR-003, BS-002 — ссылки корректны, дублирования нет.

- **LOGIC-004 (12ч)**: используется в BS-001 и SCR-005 — ссылки корректны.

### 4. Расхождения: требования ↔ design brief ↔ ТЗ

#### C1 — BS-002: трёхшаговый поток vs одноэкранный
| Источник | Описание |
|----------|----------|
| **Design brief** BS-002 | 3 шага: (1) шторка «Перенести бронь?» с текущей бронью → (2) полноэкранный выбор слота → (3) шторка подтверждения «Было/Стало» |
| **Mobile spec** BS-002 | Один полноэкранный экран: текущая бронь в хедере + список слотов. Нет шторки подтверждения, нет сравнения «Было/Стало». |

**Рекомендация:** привести mobile spec в соответствие с design brief: добавить шаг 1 и шаг 3 как отдельные состояния шторки BS-002. Либо согласовать упрощение с владельцем продукта и обновить design brief.

#### C2 — SCR-007: кнопка «Выйти» отсутствует
| Источник | Описание |
|----------|----------|
| **Design brief** SCR-007:62 | «Выйти (завершение сессии)» |
| **Mobile spec** SCR-007 | Нет элемента «Выйти», нет описания logout-действия. |

**Рекомендация:** добавить в SCR-007 элемент «Выйти»: кнопка (outline) → сброс токена (LOGIC-001) → переход на SCR-006.

#### M1 — _SCREEN_TEMPLATE.md: легаси-пути
Шаблон `_SCREEN_TEMPLATE.md` содержит:
- `rigla_network/rest/domains/{domain}.yaml` (строка 149) — должно быть `../api/{domain}/api.yaml`
- `REQ-FUNC-*` (строка 372) — должно быть `FR-*`
- `rigla_network/gql/schema_rigla.graphql` (строка 175) — GraphQL отсутствует в проекте
- Секция «Доступные спецификации» (строки 253–265) перечисляет легаси-домены (`cart_mine.yaml`, `checkout.yaml`, `wishlist.yaml` и др.)

**Рекомендация:** обновить шаблон в соответствии с соглашениями из `README.md:112-118`.

#### M2 — SCR-006: плейсхолдер в заголовке
`SCR-006_Auth.md:1` содержит `{Название экрана/шторки}` — не заменён на «Авторизация».

**Рекомендация:** заменить на `# Авторизация`.

#### M3 — SCR-001: горизонтальная полоса дат
| Источник | Описание |
|----------|----------|
| **Design brief** SCR-001:27-28 | Горизонтальная скроллируемая полоса из 7 дат: «[←] Пн 14 · Вт 15 · Ср 16 · Чт 17 [→]», с точкой под сегодня |
| **Mobile spec** SCR-001 | Только кнопка фильтра (📅) в Top App Bar. Полоса дат не упомянута. |

**Рекомендация:** добавить горизонтальную полосу дат в макет и элементы SCR-001 либо согласовать упрощение и обновить design brief.

#### M4 — FR-2.3: порог статуса «Постоянный»
| Источник | Значение |
|----------|----------|
| **FR-2.3** | «при посещении **3+** классов за последние 2 месяца» |
| **Design brief** SCR-007:43 | «8 из 12 классов до «Постоянный»» и «Постоянный клиент (**5** классов за 2 месяца)» |
| **Mobile spec** SCR-007 | Не конкретизирует число — берёт `loyaltyStatus` из API |

**Рекомендация:** заменить «5» на «3+» в design brief SCR-007:43. В mobile spec порог не нужен (статус приходит из API), но можно добавить примечание о бизнес-правиле.

#### m1 — FR-3.2: редактирование аллергий в форме бронирования
| Источник | Поведение |
|----------|-----------|
| **FR-3.2** | «автоматически подтягивать список аллергий из профиля» — только pull |
| **US-2** | «один раз настроить список в профиле» — edit in profile only |
| **Design brief** SCR-003:51-53 | Аллергии editable: «+ Добавить аллергию», чипы с ✕ |
| **Mobile spec** SCR-003:238 | Аллергии read-only: «Список аллергий из профиля (read-only)» |

Mobile spec корректнее относительно FR. **Рекомендация:** обновить design brief SCR-003 — убрать кнопку «+ Добавить аллергию» и кнопки удаления из формы бронирования. Редактирование — только в профиле (SCR-007).

#### m2 — BS-004: недостаток деталей
Design brief BS-004 содержит: месячный календарь, быстрые фильтры («Сегодня», «Завтра», «Эта неделя», «Следующая неделя»), 30-дневный диапазон, индикатор дат с классами. Mobile spec BS-004: «Календарь / список дат» — без уточнений.

**Рекомендация:** детализировать BS-004 согласно design brief.

#### m3 — SCR-004: отсутствует блок о напоминаниях
Design brief SCR-004:49: «ⓘ Напоминания придут: за 24 ч, 3 ч и 30 мин до начала (FR-5.1)». Mobile spec SCR-004: этот информационный блок отсутствует.

**Рекомендация:** добавить информационный блок о push-напоминаниях в макет SCR-004.

#### m4 — Хардкод «3 часа»
В design brief многократно используется длительность «3 часа» (SCR-001:39, SCR-002:38, BS-004 и др.). В API-модели Slot поле `duration` отсутствует — длительность подразумевается, но не параметризована.

**Рекомендация:** либо добавить `duration` в модель Slot API, либо вынести «3 часа» в константу/design-токен (не хардкодить в каждом экране).

#### m5 — Структура меню: список vs строка
Design brief SCR-002:43-45 показывает меню как структурированный список блюд. API-модель Slot:129 определяет `menu` как `string` — единую строку.

**Рекомендация:** уточнить в API, будет ли `menu` структурированным (массив строк) или плоским текстом. Если плоским — обновить design brief.
