# Onboarding проекта — «Шеф-стол»

> Документ для быстрого погружения в проект. Составлен на основе ТЗ `5-mobile-app-spec/`, модели данных `4-design/data-model.md` и контрактов API `api/`.

---

## 1. Что такое «Шеф-стол»

**Шеф-стол** — мобильное приложение (Android) для клиентов кулинарной студии. Позволяет просматривать расписание мастер-классов, бронировать места, управлять бронями (отмена, перенос), оценивать шефов и вести профиль.

- **Роль в скоупе:** только «Клиент». Шеф и администратор — вне поставки (NFR-4).
- **Платформа:** нативное Android-приложение (NFR-2).
- **Бэкенд:** black-box, вся бизнес-логика на бэкенде, приложение — потребитель REST API (NFR-3).
- **Авторизация:** только OTP по SMS (FR-2.1).
- **Оплата:** заглушка «Скоро: онлайн-оплата» в MVP. Реальный платёж — Phase 2.

## 2. Архитектура системы

```
┌──────────────────────┐       REST (Bearer token)       ┌──────────────────────┐
│   Android App        │ ◄──────────────────────────────► │       Backend         │
│   (Клиент)           │                                  │   (black-box)         │
│                      │                                  │                       │
│  - OTP Auth          │    auth/     ◄── JWT токены      │                       │
│  - Расписание        │    slots/    ◄── слоты           │                       │
│  - Бронирование      │    bookings/ ◄── брони, отзывы   │                       │
│  - Профиль           │    profile/  ◄── аллергии,       │                       │
│  - Push-уведомления  │                 экипировка       │                       │
└──────────────────────┘                                  └──────────────────────┘
```

### API-домены (5 спецификаций OpenAPI)

| Домен | Файл | Назначение |
|-------|------|------------|
| `auth` | `api/auth/api.yaml` | OTP-авторизация: `requestOtp`, `verifyOtp` |
| `slots` | `api/slots/api.yaml` | Расписание: `listSlots`, `getSlot` |
| `bookings` | `api/bookings/api.yaml` | Брони: `createBooking`, `listBookings`, `getBooking`, `cancelBooking`, `transferBooking`, `upsertReview` |
| `profile` | `api/profile/api.yaml` | Профиль клиента: `getProfile`, `updateProfile` |
| `instructors` | `api/instructors/api.yaml` | Шефы: `getInstructor` (в MVP не используется — данные идут inline в слотах) |

### Аутентификация
- Все авторизованные запросы: `Authorization: Bearer {token}`
- 401 → сброс токена → редирект на экран авторизации (LOGIC-001).

## 5. Доменные сущности

### 5.1. Client (Клиент)

Конечный пользователь приложения. Идентифицируется по номеру телефона.

| Атрибут | Тип | Доступ | Где используется |
|---------|-----|--------|------------------|
| `id` | UUID | 👁 R | Системный идентификатор |
| `phone` | string | 👁 R | SCR-006 (ввод), SCR-007 (read-only) |
| `allergies` | string[] | ✏️ W | SCR-003 (автоподстановка), SCR-007 (редактирование) |
| `ownEquipment` | boolean | ✏️ W | SCR-003 (выбор по умолчанию), SCR-007 (редактирование) |
| `loyaltyPoints` | int | 👁 R | SCR-007 (отображение баллов) |
| `loyaltyStatus` | string? | 👁 R | SCR-007 («Постоянный» / null) |

**Источник:** `GET /profile` → `PATCH /profile`
**Правила:**
- Регистрация — только через SMS на номер телефона.
- `loyaltyPoints` и `loyaltyStatus` вычисляются бэкендом, приложение не модифицирует.
- `allergies` автоматически подтягиваются в форму бронирования (SCR-003).

---

### 5.2. Instructor (Шеф)

Сотрудник студии, ведущий мастер-классы. **Полностью read-only** для приложения.

| Атрибут | Тип | Доступ | Где используется |
|---------|-----|--------|------------------|
| `id` | UUID | 👁 R | Связь Slot → Instructor |
| `name` | string | 👁 R | SCR-001, SCR-002, SCR-003, SCR-004, SCR-005, BS-001, BS-002, BS-003 |
| `status` | string | 👁 R | «Постоянный» / «Приглашённый» |
| `rating` | decimal (1.0–5.0) | 👁 R | SCR-001, SCR-002 (★ {rating}) |
| `specialization` | string? | 👁 R | Специализация (опционально) |

**Способ получения:** inline в ответах `listSlots` / `getSlot` / `listBookings?expand=slot`. Отдельный запрос `getInstructor` в MVP не используется.
**Правила:**
- Рейтинг пересчитывается бэкендом после каждого отзыва (upsertReview).
- Управление шефами — через админку (вне скоупа).

---

### 5.3. Slot (Слот / Класс)

Одно событие в расписании студии — кулинарный мастер-класс. **Полностью read-only** для приложения.

| Атрибут | Тип | Доступ | Где используется |
|---------|-----|--------|------------------|
| `id` | UUID | 👁 R | Сквозной параметр навигации: SCR-001 → SCR-002 → SCR-003 |
| `dateTime` | datetime | 👁 R | Дата/время начала (SCR-001, SCR-002, SCR-003) |
| `menu` | string | 👁 R | Название программы («Итальянская паста») |
| `photoUrls` | string[] | 👁 R | Карусель фото в SCR-002 |
| `difficulty` | string | 👁 R | «Для новичков» / «Для опытных» |
| `instructor` | object (inline) | 👁 R | Денормализованный объект шефа |
| `capacity` | int (8–12) | 👁 R | Общая вместимость |
| `bookedCount` | int | 👁 R | Занято мест |
| `price` | decimal | 👁 R | Стоимость в рублях |
| `address` | string | 👁 R | Адрес студии |
| `status` | string | 👁 R | «Активен» / «Отменён студией» |

**Источник:** `GET /slots` (список), `GET /slots/{slotId}` (детали)
**Производные поля (вычисляются клиентом — LOGIC-002):**
- `свободно = capacity − bookedCount`
- `canBook = (status == "Активен") AND (now() ≤ dateTime − 10мин) AND (свободно > 0)`

**Правила:**
- Расписание формируется бэкендом на 7 дней вперёд от запрошенной даты.
- Запись возможна не позднее 10 минут до начала.
- Если `status = "Отменён студией"` — запись заблокирована (FR-4.3), отображается бейдж «Отменён».
- `capacity` до 12 мест, до 8 — для классов с духовками.

---

### 5.4. Booking (Бронь)

Связь клиента со слотом. **Создаётся, отменяется и переносится клиентом** через API.

| Атрибут | Тип | Доступ | Где используется |
|---------|-----|--------|------------------|
| `id` | UUID | 👁 R | Идентификатор брони (SCR-004, BS-001, BS-002, BS-003) |
| `clientId` | UUID (FK) | 👁 R | Связь с Client |
| `slotId` | UUID (FK) | 👁 R | Связь со Slot |
| `createdAt` | datetime | 👁 R | Дата/время создания |
| `status` | string | 👁 R¹ | Статус (см. диаграмму состояний) |
| `equipmentType` | string | ✏️ W | «Своя» / «Прокат» |
| `refundAmount` | decimal? | 👁 R | Сумма к возврату при отмене ≤12ч |
| `reviewRating` | int? (1–5) | ✏️ W | Оценка шефа |
| `reviewText` | string? | ✏️ W | Текст отзыва |
| `slot` | object (expand) | 👁 R | Вложенный слот (при `?expand=slot`) |

> ¹ Статус меняет бэкенд; клиент инициирует переходы (отмена, перенос).

**Источник:** `POST /bookings` (создание), `GET /bookings` (список), `GET /bookings/{id}` (детали), `POST /bookings/{id}/cancel` (отмена), `POST /bookings/{id}/transfer` (перенос), `PUT /bookings/{id}/review` (оценка).

**Диаграмма состояний (статусная модель):**

```
           ┌─── cancelBooking (>12ч) ──► ОтмененаКлиентом
           │─── cancelBooking (≤12ч) ──► ОтмененаКлиентом (refundAmount = price×0.5)
Активна ──┤─── студия отменяет класс ──► ОтмененаСтудией (push)
           │─── класс состоялся ───────► Завершена (можно оценить)
           └─── no-show ───────────────► КлиентНеПришёл (начислены баллы)
```

**Правила:**
- Бэкенд гарантирует атомарность: 0 двойных броней (NFR-1).
- `equipmentType` наследуется при переносе брони.
- Оценку можно выставить и редактировать в любой момент после завершения.
- История броней — за последние 3 месяца (FR-2.4).

---

### 5.5. Review (Отзыв)

Встроенная в Booking сущность (1:0..1). Не имеет отдельного API-endpoint'а — используется upsert-семантика через `PUT /bookings/{id}/review`.

| Атрибут | Тип | Доступ |
|---------|-----|--------|
| `reviewRating` | int (1–5) | ✏️ W |
| `reviewText` | string? | ✏️ W |

**Правила:**
- Выставляется только для завершённых броней (`status = "Завершена"`).
- Редактируется в любой момент без ограничений по времени.
- После сохранения бэкенд пересчитывает `instructor.rating`.

---

### 5.6. Auth Session (Сессия)

Не бизнес-сущность, а техническая. Управляется LOGIC-001.

| Атрибут | Хранение | Описание |
|---------|----------|----------|
| `auth_token` | EncryptedSharedPreferences | JWT Bearer-токен |
| `token_expires_at` | EncryptedSharedPreferences | Timestamp истечения |

**Жизненный цикл:** `requestOtp(phone)` → SMS → `verifyOtp(phone, code)` → `token` → прикрепляется ко всем запросам → 401 → сброс → редирект на SCR-006.

---

### 5.7. Push Notification (Push-уведомление)

Системное уведомление, отправляемое бэкендом. Приложение обрабатывает два типа.

| Тип | Событие | Payload | Навигация по тапу |
|-----|---------|---------|-------------------|
| `reminder` | Напоминание (24ч/3ч/30мин) | `{ type, bookingId, title, body }` | SCR-005 → активная бронь |
| `cancel` | Отмена студией | `{ type, bookingId, title, body, reason }` | SCR-005 → отменённая бронь |

---

### 5.8. Date Filter (Состояние фильтрации)

Локальное состояние приложения, не хранится на бэкенде.

| Атрибут | Тип | По умолчанию |
|---------|-----|-------------|
| `filterFrom` | `YYYY-MM-DD` | `today()` |
| `filterTo` | `YYYY-MM-DD` | `today() + 7 дней` |

**Управляется:** SCR-001 (полоса дат + BS-004), BS-002 (шаг 2 — выбор слота).

## 6. Сводка бизнес-правил

### 6.1. Доступность записи (LOGIC-002)
```
canBook = (status == "Активен") AND (now() ≤ dateTime − 10мин) AND (capacity − bookedCount > 0)
```
Pre-check на клиенте — UX-помощник, не гарантия. Финальное решение — за бэкендом (атомарная проверка в `createBooking`).

### 6.2. Правило 12-часовой отмены (LOGIC-004)
```
remaining = slot.dateTime − now()
если remaining > 12ч  → отмена бесплатна (refundAmount = null)
если remaining ≤ 12ч → возврат 50% (refundAmount = price × 0.5)
```
Порог — **ровно 12 часов**. Бэкенд пересчитывает `remaining` на момент обработки запроса.

### 6.3. Цена и возврат (LOGIC-003)
- Цена (`slot.price`) — из ответов API, не хардкодится.
- Сумма возврата (`booking.refundAmount`) — фиксируется бэкендом, приложение только отображает.
- Фактический возврат — ручной процесс владельца (не через приложение).

### 6.4. Перенос брони (BS-002)
- Трёхшаговый поток: выбор «Перенести» → полноэкранный выбор слота → подтверждение «Было / Стало».
- `equipmentType` наследуется.
- Бэкенд выполняет одной атомарной транзакцией (отмена старой + создание новой).

### 6.5. Идемпотентность (createBooking)
Каждый тап «Подтвердить запись» генерирует новый `Idempotency-Key` (UUID). Предотвращает двойное бронирование при повторной отправке.

### 6.6. Фильтрация расписания (LOGIC-005)
- Единственный фильтр — **дата** (FR-1.2). Никаких фильтров по шефу/сложности/меню в MVP.
- Диапазон: 7 дней от выбранной даты.
- По умолчанию: `from = today()`, `to = today() + 7 дней`.

## 9. Соглашения проекта

- **API-путь:** `api/{domain}/api.yaml` → `{operationId}`
- **Идентификаторы требований:** `FR-*` (функциональные), `UC-*` (сценарии), `US-*` (пользовательские истории), `NFR-*` (нефункциональные)
- **Только REST** — GraphQL отсутствует.
- **Числа** (цена, лимиты, refundAmount) — из ответов API, не хардкод.
- **Повторяемая логика** — ссылка на `LOGIC-XXX`, без дублирования.
- **Каждый экран:** 4 состояния (loading / empty / error / success) + критерии приёмки.
- **Зона авторизации:** НЗ (неавторизованная — только SCR-006), АЗ (авторизованная — все остальные).

## 10. Связи между сущностями (ER)

```
Client ──1───0..*──► Booking ──*───1──► Slot ──*───1──► Instructor
                         │
                         └── 0..1 ──► Review (reviewRating, reviewText)
```

| Связь | Тип | Кардинальность |
|-------|-----|----------------|
| Client → Booking | has | 1 : 0..* |
| Booking → Slot | books | * : 1 |
| Slot → Instructor | taught by | * : 1 |
| Booking → Review | contains | 1 : 0..1 |
